# RFC-процесс в FinBridge

RFC (Request for Comments) — это документ, в котором инженер предлагает архитектурное изменение и собирает обратную связь от команды до начала реализации. RFC помогает избежать ситуации «реализовали три недели, а потом оказалось что архитектура неверная».

## Когда нужен RFC

RFC **обязателен** для:

- Введения новой технологии или фреймворка (язык, БД, очередь, внешний сервис).
- Изменений, затрагивающих API между сервисами или внешний API мерчантов.
- Архитектурных изменений, затрагивающих 2+ команды.
- Изменений схемы данных, которые нельзя легко откатить (удаление колонок/таблиц).
- Изменений в процессах безопасности, шифрования, хранения ключей.
- Изменений, влияющих на SLA продукта.

RFC **не нужен** для:

- Рефакторинга внутри одного сервиса без изменения поведения.
- Добавления новых эндпоинтов в существующий API (достаточно PR с review).
- Технического долга внутри команды.
- Фич внутри команды без архитектурных изменений.

Если не уверен — спроси тимлида. «Лишний» RFC лучше чем пропущенный.

## Процесс создания RFC

### Шаг 1: Draft (1–2 дня)

Создай страницу в Confluence: `Engineering / RFCs / RFC-XXX: [название]`.

Номер RFC — следующий по порядку (посмотри последний в разделе). Если создаёшь несколько RFC одновременно — зарезервируй номера в канале **#engineering**.

Шаблон RFC — в Confluence: `Engineering / RFCs / _TEMPLATE`. Основные секции:

```markdown
# RFC-XXX: [Краткое название изменения]

**Автор:** @имя.фамилия
**Дата:** ГГГГ-ММ-ДД
**Статус:** Draft | In Review | Accepted | Rejected | Deprecated

---

## Проблема

Что не так сейчас? Какие боли это создаёт?
Конкретика: цифры, примеры инцидентов, метрики.

## Цель

Чего хотим достичь после реализации?
Измеримо: «снизить p99 latency checkout с 800 мс до 300 мс»

## Предлагаемое решение

Детальное описание предлагаемого подхода.
Диаграммы (Miro), примеры кода, схемы данных.

## Альтернативы

Какие другие подходы рассматривались?
Почему они хуже предложенного?

## Влияние

- На производительность
- На надёжность
- На безопасность
- На другие команды
- На мерчантов (если меняется внешний API)

## Миграционный план

Как переходим от текущего состояния к новому?
Есть ли обратная совместимость?
Сколько времени займёт реализация?

## Открытые вопросы

Что ещё не решено и требует обсуждения?
```

### Шаг 2: Review (минимум 5 рабочих дней)

1. Напиши в **#engineering**: «RFC-XXX [название] готов к review. Жду обратной связи до [дата]. [ссылка на Confluence]».
2. Тегни людей, чьё мнение критично (тимлиды затронутых команд, Security для RFC по безопасности, DBA для RFC по схеме данных).
3. Собирай комментарии в Confluence — именно там, не в Slack. Это сохраняет историю обсуждения.
4. Отвечай на комментарии, обновляй документ.

### Шаг 3: Решение

По истечении review-периода автор фиксирует итог:

- **Accepted** — RFC принят, можно приступать к реализации. Если есть разногласия — тимлид или CTO принимают финальное решение.
- **Rejected** — RFC отклонён с объяснением. Это не приговор: можно доработать и переоткрыть.
- **Needs More Work** — требует доработки, статус остаётся Draft.

Итоговое решение фиксируется в документе и публикуется в **#engineering**.

### Шаг 4: Реализация

Accepted RFC становится эпиком в Jira. В описании эпика — ссылка на RFC. Все задачи в эпике ссылаются на RFC для контекста.

Если в процессе реализации появились значительные отклонения от RFC — обновляй RFC-документ или создавай новый.

## Примеры принятых RFC

| RFC | Тема | Год |
|---|---|---|
| RFC-012 | Декомпозиция монолита на микросервисы | 2021 |
| RFC-031 | Замена Redis Pub/Sub на RabbitMQ для очередей | 2023 |
| RFC-038 | Переход на cursor-based пагинацию во внешнем API | 2023 |
| RFC-044 | Внедрение Circuit Breaker через Istio | 2024 |
| RFC-047 | Централизованное управление секретами через Vault | 2024 |
| RFC-051 | Партиционирование таблицы payments по месяцу | 2025 |

Все RFC доступны в Confluence: `Engineering / RFCs`. Изучение принятых RFC — хороший способ понять, как эволюционировала архитектура FinBridge.

## Культура RFC

- RFC не означает что автор неправ и его заставляют защищать решение. Это коллективный процесс улучшения решений.
- Комментарии к RFC должны быть конструктивными: «это может привести к X, потому что Y» — хорошо. «Это плохая идея» — плохо.
- Молчание за 5 дней = согласие. Если у тебя есть мнение — напиши его.
- RFC не замедляет разработку — он предотвращает переделку через месяц.
