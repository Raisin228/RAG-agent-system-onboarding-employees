# Гайдлайн по Code Review в FinBridge

Code Review — обязательный этап перед мержем любого MR. Это не формальность: исторически 60% прод-инцидентов в FinBridge связаны с кодом, который прошёл поверхностное ревью. Задача ревьюера — найти проблемы до того, как они попадут в прод.

## Кто делает ревью

- Минимум **2 аппрува** от коллег из команды.
- Один из ревьюеров должен быть **senior или lead**.
- Автор MR не может аппрувить свой же MR.
- Если MR затрагивает код другой команды (например, платёжный процессинг), обязательно добавить ревьюера из этой команды.

Назначение ревьюеров — вручную в GitLab. Автоматическое назначение через `CODEOWNERS`:

```
# .gitlab/CODEOWNERS
/payments/         @payments-team-leads
/api-gateway/      @platform-team-leads
/fraud/            @fraud-team-leads
```

## Временные рамки

| Приоритет MR | Ожидаемое время первого ревью |
|---|---|
| 🔥 HOTFIX | 30 минут (пишем в #engineering с упоминанием) |
| Обычный | 1 рабочий день |
| Рефакторинг / chore | 2 рабочих дня |

Если ревьюер видит MR в очереди больше дня — напомни ему в Slack напрямую. Молча ждать нельзя.

## Что проверяет ревьюер

### Корректность логики

- Соответствует ли код описанию задачи в Jira?
- Обрабатываются ли граничные случаи (пустые значения, нулевые суммы, истёкшие токены)?
- Нет ли race condition в конкурентном коде?
- Все ли ошибки правильно обрабатываются и логируются?

### Производительность

- Нет ли N+1 запросов к БД? (типичная ошибка: загрузка сущностей в цикле вместо JOIN)
- Нет ли запросов без индексов на больших таблицах? Для таблиц с >1M строк — обязательно проверить `EXPLAIN ANALYZE`.
- Не создаются ли тяжёлые объекты внутри hot path (цикл, обработчик каждого запроса)?

### Безопасность

- Нет ли SQL-инъекций (параметризованные запросы)?
- Нет ли открытой передачи пользовательского ввода в системные вызовы?
- Не возвращает ли API лишние данные (чужие платёжные данные, внутренние ID)?
- Не хардкодятся ли секреты? (см. `secrets_management.md`)

### Читаемость и поддерживаемость

- Понятны ли имена переменных, функций, типов?
- Нет ли функций длиннее 80 строк (исключение — явно обоснованный случай)?
- Написаны ли тесты к новой логике? (обязательно, см. `testing_guidelines.md`)
- Есть ли комментарии к нетривиальным решениям?

## Как оставлять комментарии

Используй префиксы, принятые в FinBridge:

| Префикс | Смысл | Блокирует мерж? |
|---|---|---|
| `[BLOCKER]` | Критичная ошибка, мерж запрещён | Да |
| `[MAJOR]` | Важная проблема, желательно исправить | По усмотрению ревьюера |
| `[MINOR]` | Небольшое улучшение, стиль | Нет |
| `[NIT]` | Мелочь, можно проигнорировать | Нет |
| `[QUESTION]` | Вопрос, хочу понять логику | Нет |
| `[SUGGESTION]` | Альтернативный подход, не настаиваю | Нет |

Пример хорошего комментария:
```
[BLOCKER] Здесь не обрабатывается ошибка от Vault при получении ключа шифрования.
Если Vault недоступен, транзакция упадёт с паникой, а не с корректным 503.
Предлагаю вернуть error и обработать его в вызывающем коде.
```

Пример плохого комментария:
```
Это неправильно
```

## Обязанности автора MR

- **Описание MR** должно объяснять *что* сделано и *почему*, а не просто «реализована задача PAY-12345».
- **Саморевью** — перед назначением ревьюеров пройдись по своему diff сам. Убери отладочные `fmt.Println`, закомментированный код, случайные файлы.
- **Отвечай на комментарии** в течение рабочего дня. Если комментарий понял и исправил — нажми **Resolve thread**. Если не согласен — аргументируй письменно.
- **Не закрывай MR молча** — если передумал, напиши причину в описании.

## Правило двух дней

Если MR без изменений висит больше 2 рабочих дней (нет ни аппрувов, ни комментариев) — пиши тимлиду. Ревью не должно быть узким местом.

## Автоматические проверки (CI)

До того как ревьюер увидит MR, CI уже проверит:

```
Lint (golangci-lint / eslint)       — стиль и типичные ошибки
Unit тесты                          — покрытие не ниже 70% для изменённых файлов
SAST (semgrep)                      — уязвимости
Dependency scan (trivy)             — CVE в зависимостях
Secret scan (gitleaks)              — случайно закоммиченные секреты
Build                               — проект собирается без ошибок
```

Если что-то упало — ревьюер имеет право не смотреть MR до зелёного CI. Это норма.

По вопросам — **#engineering** или тимлид.
