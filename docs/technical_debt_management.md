# Управление техническим долгом в FinBridge

Технический долг — это неизбежная часть разработки продукта. В FinBridge у нас системный подход к его управлению: долг виден, приоритизирован и постепенно выплачивается — вместо того чтобы накапливаться и взрываться в самый неподходящий момент.

## Определения

**Технический долг** — всё, что усложняет разработку или снижает надёжность системы:
- Устаревшие зависимости с известными CVE
- Код, который сложно понять и изменить (запутанная логика, нет тестов)
- Архитектурные решения, которые были правильными тогда, но тормозят сейчас
- Дублирование логики между сервисами
- Нарушение собственных стандартов (например, секрет в .env вместо Vault)

**Не технический долг:** задача которую не успели, нерабочая фича, баг в бизнес-логике — это обычные задачи.

## Где фиксируется технический долг

Все элементы технического долга — в Jira, проект **PAY**, тип задачи **Tech Debt**. Это обеспечивает видимость для продактов и менеджеров.

### Обязательные поля при создании тикета

- **Заголовок**: конкретный, с названием компонента. «Рефакторинг ProcessPayment» — плохо. «payments-core: убрать глобальный mutex в ProcessPayment, заменить на per-payment locking» — хорошо.
- **Описание**: почему это технический долг? Какой ущерб он наносит прямо сейчас? (медленнее разработка, падает надёжность, риск безопасности)
- **Impact**: оценка влияния по шкале: `low` / `medium` / `high` / `critical`.
- **Effort**: оценка трудозатрат: `XS` (< 2ч) / `S` (0.5 дня) / `M` (1–3 дня) / `L` (неделя) / `XL` (> недели).
- **Component**: к какому сервису относится.

## Квота на технический долг

Каждая команда тратит **20% времени каждого спринта** на технический долг. Это не обсуждается и не убирается под давлением бизнеса — это правило Platform Engineering и Engineering Manager.

Механизм: при планировании спринта в Jira у команды есть виртуальный бюджет в story points. 20% этого бюджета автоматически резервируется для Tech Debt тикетов.

Если в данном спринте нет горящего технического долга — 20% идут на обучение, исследования или улучшение тулинга.

## Приоритизация

Каждый квартал тимлид с командой проводит **Tech Debt Grooming** (1.5–2 часа):

1. Просматривают все открытые Tech Debt тикеты.
2. Закрывают уже неактуальные.
3. Приоритизируют по матрице Impact × Effort.
4. Топ-5 тикетов помечаются `tech-debt-priority-now` — они попадают в ближайший спринт.

| Impact \ Effort | XS/S | M | L/XL |
|---|---|---|---|
| Critical | Делаем немедленно | Делаем в этом спринте | Дробим на части |
| High | Делаем в этом спринте | Приоритет в следующем | Планируем |
| Medium | По возможности | Очередь | Backlog |
| Low | Backlog | Backlog | Возможно никогда |

## Как создать тикет на технический долг

```bash
# Нашёл проблему в коде — создай тикет прямо из IDE
# Плагин Jira для VS Code: Ctrl+Shift+P → "Jira: Create Issue"

# Или через CLI jira
jira create \
  --project PAY \
  --type "Tech Debt" \
  --summary "payments-core: remove deprecated provider_v1 adapter" \
  --description "provider_v1 was deprecated in rc/25.800, now all traffic on provider_v2. Dead code increases cognitive load and test time." \
  --priority Medium \
  --label tech-debt \
  --component payments-core
```

Или просто открой `jira.finbridge.io` → Create Issue.

## TODO-комментарии в коде

TODO в коде **обязательно** должны содержать ссылку на Jira-тикет:

```go
// ✅ Правильно:
// TODO(PAY-98765): replace mutex with per-payment channel to eliminate contention
// See: https://jira.finbridge.io/browse/PAY-98765
mu.Lock()
defer mu.Unlock()

// ❌ Неправильно:
// TODO: fix this later
// TODO: refactor
// FIXME: this is hacky
```

CI проверяет что все TODO содержат формат `TODO(PAY-XXXXX)`. Если не нашёл — создай тикет за 2 минуты.

## Arch Debt — архитектурный долг

Для крупных архитектурных проблем (масштаб: месяцы работы, несколько команд) используется процесс **RFC**:

1. Автор описывает проблему и предлагаемое решение в Confluence: `Engineering / RFCs / RFC-XXX`.
2. RFC обсуждается в **#engineering** минимум 5 рабочих дней.
3. Принятые RFC превращаются в эпики в Jira с разбивкой на задачи.

Примеры решённого архитектурного долга:
- `RFC-012`: миграция с монолита на микросервисы (2021–2022)
- `RFC-031`: замена самописной очереди на RabbitMQ (2023)
- `RFC-047`: переход на Vault для управления секретами (2024)

Текущие открытые RFC: Confluence → `Engineering / RFCs` → фильтр `Status: In Review`.
