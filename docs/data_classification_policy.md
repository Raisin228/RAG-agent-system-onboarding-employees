# Классификация данных в FinBridge

FinBridge работает с чувствительными финансовыми данными и обязан соблюдать требования PCI DSS Level 1, 152-ФЗ (персональные данные) и внутренней политики безопасности. Понимание того, какие данные как можно использовать — обязанность каждого сотрудника.

## Уровни классификации

### Уровень 1: Public (Публичный)

Данные, предназначенные для открытого доступа. Раскрытие не несёт ущерба.

**Примеры:** публичная документация API, маркетинговые материалы, пресс-релизы, открытый исходный код.

**Правила использования:** без ограничений.

---

### Уровень 2: Internal (Внутренний)

Данные для внутреннего использования сотрудниками FinBridge. Не предназначены для внешних лиц, но утечка не критична.

**Примеры:** внутренние процессы, Confluence-документация, переписка в Slack, метрики продукта (не финансовые), организационная структура.

**Правила использования:**
- Не передавать третьим лицам без согласования с руководством.
- Хранить только на корпоративных системах (не в личных Google Drive, Dropbox и т.д.).
- Можно использовать на личных устройствах при соблюдении `security_policy.md`.

---

### Уровень 3: Confidential (Конфиденциальный)

Бизнес-чувствительные данные. Утечка может нанести репутационный или финансовый ущерб.

**Примеры:** финансовые показатели компании, клиентские данные (кроме платёжных), стратегические планы, коммерческие условия с партнёрами, исходный код ключевых компонентов, данные HR.

**Правила использования:**
- Доступ — только сотрудникам с производственной необходимостью (need-to-know).
- Не обсуждать в публичных местах (кафе, транспорт) где могут услышать.
- Не передавать по незащищённым каналам (незашифрованная почта, публичные мессенджеры).
- При передаче контрагентам — только с NDA и через защищённые каналы.
- Хранить только в корпоративных системах с контролем доступа.

---

### Уровень 4: Restricted (Строго ограниченный / PCI DSS)

Данные, подпадающие под требования PCI DSS и 152-ФЗ. Максимальный уровень защиты.

**Примеры:**
- **Cardholder Data (CHD):** полный номер карты (PAN), имя держателя, срок действия, CVV/CVC
- **Sensitive Authentication Data (SAD):** полная магнитная полоса, PIN-блок
- **Персональные данные клиентов:** паспортные данные, ИНН, СНИЛС, дата рождения
- **Ключи шифрования и сертификаты**
- **Реквизиты банковских счетов клиентов**

**Правила использования — СТРОГО:**
- Доступ только в рамках PCI DSS scope (специально выделенные системы).
- **Нельзя** хранить CVV/CVC после авторизации транзакции (ни в каком виде).
- **Нельзя** выводить в логи (даже замаскированные — только последние 4 цифры PAN разрешены).
- **Нельзя** копировать на dev/staging окружения. На тестовых стендах — только синтетические данные.
- **Нельзя** хранить на личных устройствах, внешних носителях.
- **Нельзя** передавать по email, Slack или любым незашифрованным каналам.
- При работе с production данными — только через специальный approval process (тикет в Jira, проект SEC).

## Синтетические данные для тестирования

Для всех тестов используются **только синтетические данные**. Генераторы:

```bash
# Генерация тестовых карточных данных (Luhn-valid, но не реальные)
fb-testdata generate --type payment_card --count 100

# Генерация тестовых мерчантов
fb-testdata generate --type merchant --count 50

# Загрузить синтетический датасет в staging БД
fb-testdata seed --env staging --scenario checkout_flow
```

Синтетические номера карт в тестовых средах:
- Visa: `4111111111111111`
- Mastercard: `5500005555555559`
- МИР: `2201000000000000`

## Обращение с данными в коде

```go
// ПРАВИЛЬНО — храним только маскированные данные в логах
log.Info("payment authorized",
    zap.String("pan_last4", pan[len(pan)-4:]),  // только последние 4 цифры
    zap.String("payment_id", payment.ID),
)

// НЕПРАВИЛЬНО — полный PAN в логах
log.Info("payment authorized",
    zap.String("pan", pan),  // ❌ нарушение PCI DSS
)

// ПРАВИЛЬНО — шифруем PAN перед хранением
encryptedPAN, err := vault.Encrypt(ctx, pan)  // через Vault Transit
// Храним только encryptedPAN, исходный pan нигде не сохраняем

// НЕПРАВИЛЬНО — CVV нельзя хранить вообще, даже зашифрованным
payment.CVV = cvv  // ❌ CVV нельзя хранить после авторизации
```

## Нарушения и последствия

Нарушение правил обращения с данными Уровня 4 (PCI DSS / ПДн) — основание для немедленного расследования и может повлечь:
- Дисциплинарные меры вплоть до увольнения.
- Персональную юридическую ответственность (в рамках 152-ФЗ — до административной/уголовной).
- Штрафы для компании от регуляторов (Банк России, Роскомнадзор) до 18 млн ₽.
