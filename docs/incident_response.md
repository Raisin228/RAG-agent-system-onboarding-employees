# Реагирование на инциденты в FinBridge

Инцидент — это любое незапланированное событие, которое нарушает работу продукта или создаёт риск для данных клиентов. В FinBridge обрабатывается 12 миллионов транзакций в месяц, поэтому даже 10 минут простоя — это тысячи неудавшихся платежей и прямые убытки мерчантов.

## Severity-уровни

| Уровень | Описание | Примеры | Цель по времени восстановления (RTO) |
|---|---|---|---|
| **SEV-1** | Полный сервисный сбой, прод недоступен | Процессинг не принимает транзакции, API-gateway не отвечает | 30 минут |
| **SEV-2** | Частичный сбой, критический сценарий не работает | Отказ одного платёжного провайдера, задержки > 5 сек на checkout | 2 часа |
| **SEV-3** | Деградация, workaround есть | Медленные отчёты, единичные ошибки у мерчантов | 8 часов |
| **SEV-4** | Незначительная проблема | Косметический баг в дашборде, неверная локализация | Следующий спринт |

## Кто дежурит

Дежурный инженер (on-call) назначается еженедельно. Расписание — в документе `on_call_rotation.md`. Смотреть текущего дежурного:

```
/oncall who     # в любом Slack-канале, бот ответит
```

Дежурный получает алерты через **PagerDuty**. Если дежурный не ответил на звонок PagerDuty в течение 5 минут — эскалация на тимлида. Если тимлид не ответил ещё через 5 минут — эскалация на CTO Артёма Красильникова (@artem.krasilnikov).

## Алгоритм действий при инциденте

### Шаг 1. Обнаружение и объявление (первые 5 минут)

1. Увидел алерт в **#incidents** или получил звонок от PagerDuty — немедленно откликнись.
2. В канале **#incidents** напиши `@oncall инцидент принят` и укажи severity.
3. Создай тред инцидента по шаблону:

```
🔴 SEV-1 INCIDENT — [краткое описание]

Время начала: HH:MM МСК
IC (Incident Commander): @имя
Симптомы: [что происходит, что видят пользователи]
Первичная гипотеза: [если есть]
Статус: РАССЛЕДОВАНИЕ
```

Incident Commander (IC) — тот, кто принял инцидент. Он координирует работу, пока технари исследуют.

### Шаг 2. Митигация (первые 30 минут для SEV-1)

Цель — прекратить ущерб, не обязательно найти корневую причину.

Быстрые инструменты:

```bash
# Откатить последний деплой сервиса
fb-deploy rollback --service payments-core --env prod

# Включить feature flag для аварийного режима (отключение тяжёлых фич)
fb-flags set payments.enable_fraud_check=false --env prod

# Масштабировать под нагрузку
kubectl scale deployment payments-core --replicas=20 -n prod
```

Список сервисов, их зависимости и процедуры откатки — в Confluence: `Runbooks / Services / <service-name>`.

### Шаг 3. Коммуникация

- Каждые **15 минут** для SEV-1, **30 минут** для SEV-2 — обновляй тред в **#incidents**.
- Если SEV-1 длится более 30 минут — напиши в **#general** краткий статус для всей компании.
- Если затронуты мерчанты — аккаунт-менеджеры пишут им. Ты сообщаешь аккаунт-менеджерам (@account-managers в Slack) когда можно оповещать и что говорить.

### Шаг 4. Восстановление и верификация

1. После применения фикса или откатки убедись, что метрики вернулись к норме.
2. Проверь ключевые SLI в Grafana: дашборд **"Payments Core — SLI Overview"** (`grafana.finbridge.io/d/payments-sli`).

Нормальные показатели:
- Error rate: < 0.1%
- p99 latency checkout: < 800 мс
- Throughput: > 200 TPS

3. Напиши в **#incidents**: `✅ Инцидент закрыт. Сервис работает нормально. RCA — в течение 48 часов.`

### Шаг 5. Постмортем (Post-Mortem)

Для SEV-1 и SEV-2 — **обязателен** в течение 48 часов. Для SEV-3 — желателен.

Шаблон постмортема — в Confluence: `Engineering / Postmortems / Template`. Секции:

1. **Summary** — одним абзацем что произошло
2. **Timeline** — хронология событий с временными метками
3. **Root Cause** — корневая причина (5 WHYs)
4. **Impact** — сколько транзакций затронуто, SLA-breach, мерчанты
5. **What went well** — что сработало хорошо
6. **What went wrong** — что нужно улучшить
7. **Action Items** — конкретные задачи в Jira с ответственными и дедлайнами

Культура постмортемов в FinBridge — **blameless**. Мы ищем системные проблемы, а не виновных. Называть имена в негативном ключе в постмортеме запрещено.

## Полезные команды для диагностики

```bash
# Логи сервиса в проде за последние 15 минут
kubectl logs -n prod -l app=payments-core --since=15m | grep -i error

# Метрики конкретного пода
kubectl top pods -n prod -l app=payments-core

# Статус всех деплойментов
kubectl get deployments -n prod

# Проверить статус платёжных провайдеров
fb-providers status --env prod

# История деплоев сервиса
fb-deploy history --service payments-core --env prod --last=10
```
