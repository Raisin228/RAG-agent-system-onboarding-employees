# Процесс планирования спринтов в FinBridge

В FinBridge используется **Scrum** с двухнедельными спринтами. Спринт начинается в понедельник и заканчивается в пятницу через две недели.

## Цикл спринта

```
Пн (нед. 1)   →   Пт (нед. 2)   →   Пн (нед. 3)
Sprint Start         Sprint End         Next Sprint Start
    │                    │                    │
    │  ←─ 10 рабочих дней ─→  │
    │                    ├─ Ретро (Пт 16:00)
    │                    └─ Planning следующего (Пт 17:00)
    └─ Daily Standup (каждый день)
```

## Встречи спринта

### Daily Standup

- **Когда:** каждый рабочий день, расписание у каждой команды своё (обычно 10:00–10:30).
- **Формат:** 15 минут, стоя (или в Zoom), строго по формату:
  1. Что сделал вчера?
  2. Что планирую сегодня?
  3. Есть ли блокеры?
- **Правило:** не обсуждать детали технических решений на стендапе. Если нужно обсудить — договориться встретиться отдельно после.

### Sprint Review (Demo Day)

- **Когда:** последняя пятница каждого месяца, 16:00–17:30 МСК, конференц-зал «Эльбрус» или Zoom.
- **Что:** каждая команда показывает что сделала за спринт. Живая демонстрация на staging — не слайды.
- **Присутствуют:** все инженеры, продакты, часть руководства, иногда клиенты.
- **Подготовка:** по 5–7 минут на команду. Назначь докладчика ротационно.

### Sprint Retrospective

- **Когда:** последняя пятница спринта, 16:00–17:00 (за час до Review).
- **Участники:** только команда + тимлид.
- **Формат:** Start / Stop / Continue:
  - **Start:** что стоит начать делать?
  - **Stop:** от чего стоит отказаться?
  - **Continue:** что работает и нужно продолжать?
- Ретро фасилитирует тимлид. Итоги фиксируются в Confluence.

### Sprint Planning

- **Когда:** понедельник первого дня нового спринта, 10:00–12:00.
- **Участники:** вся команда + продакт.
- **Порядок:**
  1. Продакт представляет top-приоритеты из бэклога (30 мин).
  2. Команда оценивает задачи в Story Points (числа Фибоначчи: 1, 2, 3, 5, 8, 13).
  3. Команда берёт задачи на спринт исходя из velocity (~80% от исторической средней).
  4. 20% мощности резервируется на Tech Debt (см. `technical_debt_management.md`).
  5. Формируется Sprint Goal — одно предложение, что команда достигнет за спринт.

## Работа с Jira

### Статусы задачи

```
Backlog → Selected for Sprint → In Progress → Code Review → QA → Done
```

| Статус | Смысл |
|---|---|
| Backlog | Задача существует, ещё не в спринте |
| Selected for Sprint | Задача взята в спринт, ещё не начата |
| In Progress | Активная разработка |
| Code Review | MR создан, ждёт аппрув |
| QA | Замержено, QA проверяет на staging |
| Done | QA принял, задача завершена |

**Правило:** статус задачи в Jira должен отражать реальность. Не оставляй задачи в «In Progress» на несколько дней без обновления — это сигнал блокера.

### Story Points

Оценка в SP — оценка сложности, не времени. Ориентир:

| SP | Примерная сложность |
|---|---|
| 1 | Тривиальное изменение, всё понятно |
| 2 | Небольшая задача, есть лёгкие неизвестные |
| 3 | Стандартная задача на 0.5–1 день |
| 5 | Задача с несколькими компонентами, 1–2 дня |
| 8 | Сложная задача с неизвестными, > 2 дней |
| 13 | Слишком большая, надо разбить |

Если задача оценивается в 13 — она дробится на подзадачи до планирования.

### Velocity

Velocity команды (среднее количество SP за спринт за последние 5 спринтов) отображается в Jira: `Reports → Velocity Chart`. Не берите больше velocity — это приводит к незакрытым задачам в конце спринта.

## Бэклог

Бэклог ведёт **Product Owner**. Разработчики могут добавлять задачи в бэклог (особенно Tech Debt), но приоритизирует их продакт.

**Backlog grooming** — раз в спринт (обычно среда второй недели, 1 час). На нём продакт объясняет задачи, команда задаёт вопросы и предварительно оценивает.

## Чего не делать

- **Не берись за задачи, которых нет в спринте**, без согласования с тимлидом — это ломает планирование.
- **Не закрывай задачу в Done самостоятельно** — статус Done ставит QA после проверки.
- **Не тяни задачу молча** если застрял больше дня — подними блокер на стендапе или напиши тимлиду.
- **Не переносить задачи из спринта в спринт** больше одного раза без объяснения причины — это сигнал о неверной оценке или скрытом блокере.
