# Стандарты документации в FinBridge

Хорошая документация — часть работы инженера, не опциональная добавка. В FinBridge принято правило: код без документации не считается завершённым для задач, которые затрагивают новые сервисы, API или архитектурные решения.

## Где хранится документация

| Тип документации | Инструмент | URL |
|---|---|---|
| Техническая документация, RFC, runbook-и | Confluence | `confluence.finbridge.io` |
| API документация (внешняя) | OpenAPI → developer.finbridge.io | `developer.finbridge.io` |
| Онбординг-документы (этот документ) | GitLab, `/docs` в репо | `gitlab.finbridge.io/finbridge/finbridge-core/docs` |
| Архитектурные диаграммы | Miro | `miro.com` (ссылки в Confluence) |
| ADR (Architecture Decision Records) | Confluence, раздел ADR | `confluence.finbridge.io/display/ENG/ADR` |

## Структура Confluence

```
Engineering/
├── Architecture/
│   ├── Services/         ← описание каждого микросервиса
│   ├── ADR/              ← Architecture Decision Records
│   └── RFCs/             ← Request for Comments (предложения изменений)
├── Runbooks/             ← инструкции по диагностике и устранению проблем
│   ├── _TEMPLATE         ← шаблон runbook-а
│   └── payments-core/
│       ├── PaymentErrorRateHigh
│       └── DatabaseConnectionPoolExhausted
├── Postmortems/          ← разборы инцидентов
│   └── _TEMPLATE
├── Onboarding/           ← материалы для новых сотрудников
└── Offboarding/          ← документы передачи дел
```

## Когда нужно писать документацию

### Обязательно

- **Новый микросервис**: страница в `Engineering / Architecture / Services / <service-name>`. Включает: назначение, ключевые сценарии, технологии, зависимости, точки конфигурации, runbook.
- **Новый API**: обновление OpenAPI-спецификации (в коде, генерируется автоматически на сайт).
- **Архитектурное решение**: ADR (см. ниже).
- **Изменение процесса**: обновить соответствующий документ в `/docs` в репозитории (этот файл — пример).
- **Инцидент SEV-1 / SEV-2**: постмортем в течение 48 часов.
- **Нетривиальный runbook**: если при инциденте пришлось делать что-то нестандартное — запиши.

### Желательно

- Новый алгоритм или нетривиальная бизнес-логика — комментарий в коде или Confluence-страница.
- Решение, которое выглядит неочевидно — объясни «почему» (не «что», это видно из кода).

## ADR — Architecture Decision Records

ADR фиксирует важное архитектурное решение: контекст, варианты, выбранный путь и последствия.

**Когда писать ADR:**
- Выбор новой технологии или замена существующей.
- Изменение способа коммуникации между сервисами.
- Изменение схемы данных с долгосрочными последствиями.
- Отказ от ранее принятого решения.

**Шаблон ADR (Confluence: `Engineering / ADR / _TEMPLATE`):**

```markdown
# ADR-042: Замена Redis Pub/Sub на RabbitMQ для webhook-очереди

## Статус
Принято (2026-03-15)

## Контекст
Webhook Dispatcher использует Redis Pub/Sub для очереди исходящих событий.
При рестарте dispatcher-а сообщения теряются — Redis Pub/Sub не персистентный.
За Q1 2026 потеряно ~12 000 webhook-событий, мерчанты не получили подтверждения.

## Варианты
1. Redis Streams (персистентные, но ограниченный replay)
2. RabbitMQ с quorum queues (уже есть в инфраструктуре, надёжный, replay через DLQ)
3. Kafka (избыточен для нашего объёма, дополнительная операционная нагрузка)

## Решение
RabbitMQ. Уже используем его для других очередей (payments.events), команда знает операционку.

## Последствия
+ Гарантированная доставка, replay при ошибках
+ Нет новой инфраструктуры
- Webhook Dispatcher нужно переписать под AMQP (оценка: 5 дней)
- Нужна миграция данных из старой Redis-очереди
```

## Как писать runbook

Runbook — практическая инструкция для on-call инженера при срабатывании алерта. Хороший runbook позволяет инженеру, незнакомому с сервисом, решить типовую проблему за 15 минут.

Структура (шаблон: `Engineering / Runbooks / _TEMPLATE`):

1. **Алерт**: название и текст алерта который триггерит этот runbook.
2. **Влияние**: что видят пользователи/мерчанты когда этот алерт срабатывает.
3. **Диагностика**: конкретные команды в правильном порядке:
   ```bash
   # 1. Проверить состояние подов
   kubectl get pods -n prod -l app=payments-core

   # 2. Посмотреть последние ошибки
   kubectl logs -n prod -l app=payments-core --since=15m | grep ERROR | tail -50
   ```
4. **Решения**: для каждого типичного root cause — шаги исправления.
5. **Эскалация**: кому звонить если стандартные шаги не помогли.

**Обновляй runbook после каждого инцидента** — если решил проблему иначе чем написано, или нашёл новый тип проблемы, дополни документ.

## Правила оформления в Confluence

- Заголовок страницы: кратко и конкретно. «Сервис X» — плохо. «payments-core: architecture and runbooks» — хорошо.
- Используй шаблоны — они в Confluence в каждом разделе.
- Выделяй команды и код в `code block`.
- Указывай дату последнего обновления в начале страницы: `_Обновлено: 2026-04-09, @bs.atroshenko_`.
- Устаревшие страницы — помечай меткой `OUTDATED` и указывай актуальную ссылку.
- Не создавай страницы в `Personal Space` — только в `Engineering/` или соответствующем разделе.

По вопросам документирования — **#engineering** или тимлид.
