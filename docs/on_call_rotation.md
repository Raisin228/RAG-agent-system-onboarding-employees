# Дежурство (On-Call) в FinBridge

Дежурный инженер — первая линия реагирования на инциденты в проде. Дежурство — это часть работы в FinBridge, обязанность каждого инженера уровня Middle и выше.

## Кто дежурит

Дежурных два:

| Роль | Обязанности |
|---|---|
| **Primary on-call** | Первый получает алерты PagerDuty, ведёт инцидент |
| **Secondary on-call** | Backup: получает эскалацию если Primary не ответил за 5 минут |

В каждой команде своё расписание. Сейчас в FinBridge 5 продуктовых команд, каждая дежурит по своим сервисам:

| Команда | Сервисы |
|---|---|
| Payments Core | payments-core, checkout-api, payment-router |
| Fraud & Risk | fraud-detector, risk-scorer, blocklist-service |
| Merchant Platform | merchant-dashboard, merchant-api, onboarding-service |
| Platform Engineering | infra, k8s, databases, CI/CD |
| Integrations | provider-adapters, webhook-dispatcher, reconciliation |

## График дежурств

Смена: **1 неделя**, с понедельника 10:00 МСК до следующего понедельника 10:00 МСК.

Расписание на квартал вперёд — в PagerDuty: `finbridge.pagerduty.com/schedules`. Там же можно поменяться с коллегой — через **Schedule Overrides**.

Текущий дежурный:
```
/oncall who        — в любом Slack-канале
/oncall who payments  — дежурный конкретной команды
```

## Обязанности дежурного

### В рабочее время (10:00–19:00 МСК)

- Отвечать на алерты PagerDuty в течение **5 минут**.
- Следить за каналом **#incidents** и **#alerts-warning**.
- Обрабатывать SEV-1 и SEV-2 инциденты — подробнее в `incident_response.md`.
- Проверять ключевые дашборды утром (занимает 5–10 минут):
  - `grafana.finbridge.io/d/payments-sli` — статус обработки платежей
  - `grafana.finbridge.io/d/platform-overview` — состояние кластера

### В нерабочее время (19:00–10:00 МСК, выходные)

- Отвечать только на **SEV-1 алерты** (PagerDuty помечает severity).
- Цель — не решить проблему полностью, а **прекратить ущерб**: откат, включение circuit breaker, масштабирование.
- Задокументировать в **#incidents** что произошло и какие шаги предприняты — даже если разбудили в 3 ночи.

## Компенсация за дежурство

| Период | Компенсация |
|---|---|
| Неделя дежурства (дополнительная нагрузка) | +15% к окладу за неделю |
| Ночной вызов (19:00–08:00) | 3 000 ₽ за каждый вызов с активными действиями > 15 мин |
| Вызов в выходной день | 5 000 ₽ за каждый вызов с активными действиями > 15 мин |

Компенсации начисляются автоматически по данным PagerDuty в следующей зарплате. Если вызов был, но PagerDuty не зафиксировал — напиши @marina.sokolova (HR).

## Как готовиться к дежурству

За день до начала смены:

1. **Проверь состояние сервисов** — нет ли известных проблем или запланированных деплоев.
2. **Прочитай постмортемы** за последний месяц своей команды (Confluence: `Engineering / Postmortems`).
3. **Убедись что PagerDuty приложение установлено** на телефоне и уведомления включены.
4. **Проверь runbook-и** для своих сервисов — актуальны ли они?

## Runbook-и

Runbook — инструкция по диагностике и решению типовых проблем сервиса. Находятся в Confluence: `Engineering / Runbooks / <service-name>`.

Каждый runbook содержит:
- Описание алерта
- Что проверить первым делом
- Команды диагностики
- Варианты митигации
- Контакты (кому звонить если ситуация нестандартная)

**Обновлять runbook после каждого инцидента** — это обязанность дежурного. Если решил проблему новым способом — запиши его.

Создать новый runbook: скопируй шаблон из `Engineering / Runbooks / _TEMPLATE`.

## Передача смены

В конец пятницы / перед заменой дежурный пишет **Handover Note** в канале **#on-call-handover**:

```
📋 Handover Note — Payments Core — смена до ПН 10:00

Статус сервисов: ✅ Всё зелёное

Инциденты за неделю:
- SEV-2 в среду 14:30: задержка у провайдера Тинькофф, mitigation — переключение трафика на СБП. Постмортем: <ссылка>

На что обратить внимание:
- Деплой 26.100 запланирован на вторник 11:00 — проверь дашборды после
- Таблица payment_events приближается к 500GB, DBA уже знает (@dmitry.kharchenko)

Открытые инциденты: нет
Открытые тикеты в работе: PAY-124500 (мониторинг решения)
```

## Инструменты

```bash
# Статус всех сервисов в проде
kubectl get deployments -n prod

# Посмотреть последние алерты (через CLI Alertmanager)
amtool alert query --alertmanager.url=https://alertmanager.finbridge.io

# Масштабировать сервис
kubectl scale deployment payments-core --replicas=15 -n prod

# Быстрый откат
fb-deploy rollback --service payments-core --env prod

# Посмотреть логи за последние 30 минут
kubectl logs -n prod -l app=payments-core --since=30m --tail=200
```
