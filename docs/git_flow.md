# Git Flow в FinBridge

В FinBridge используется релизная модель ветвления. Весь код хранится в GitLab: `gitlab.finbridge.io`. Основная единица планирования — **веха (milestone)**, привязанная к конкретному релизу продукта. За год выходит примерно 10 релизов.

## Структура веток

### Релизные ветки (Release Candidate)

Релизные ветки — это ключевые точки, к которым привязаны задачи из Jira. Именование:

```
rc/<ГОД>.<НОМЕР_ВЕРСИИ>
```

Примеры:
- `rc/26.100` — первый релиз 2026 года
- `rc/26.200` — второй релиз 2026 года
- `rc/26.300` — третий релиз

Номер версии увеличивается с шагом 100, чтобы оставить пространство для хотфиксов (например, `rc/26.101`, `rc/26.102`).

Релизные ветки создаёт релиз-менеджер. Разработчики не создают и не удаляют релизные ветки самостоятельно.

### Ветки задач (feature / bugfix / hotfix)

Каждый разработчик работает в своей ветке, которая создаётся от релизной. Формат именования:

```
<веха>/<тип>/<логин_разработчика>/<номер_задачи_Jira>
```

Где `<тип>` — один из:
- `feature` — новая функциональность
- `bugfix` — исправление бага
- `hotfix` — срочное исправление на проде
- `refactor` — рефакторинг без изменения поведения
- `chore` — технический долг, обновление зависимостей

Примеры:
- `26.100/feature/bs.atroshenko/PAY-1238765`
- `26.100/bugfix/iv.petrov/PAY-1240012`
- `26.200/refactor/an.sidorov/PAY-1251000`
- `26.100/hotfix/bs.atroshenko/PAY-1239001`

## Рабочий процесс разработчика

### 1. Получить задачу

Задача в Jira привязана к конкретной вехе (например, `26.100`). Веха — это и есть релизная ветка, в которую должен попасть твой код.

### 2. Переключиться на релизную ветку и обновить её

```bash
git checkout rc/26.100
git pull origin rc/26.100
```

**Обязательно** сделай `pull` перед созданием своей ветки. Если ты отпочкуешься от устаревшей версии — потом будут конфликты при мерже.

### 3. Создать свою ветку

```bash
git checkout -b 26.100/feature/bs.atroshenko/PAY-1238765
```

### 4. Написать код и закоммитить

Формат коммит-сообщений:

```
[PAY-1238765] Краткое описание что сделано

Подробное описание, если нужно. Почему выбран такой подход,
какие альтернативы рассматривались, на что обратить внимание при ревью.
```

Номер задачи Jira в квадратных скобках **обязателен** — GitLab автоматически привяжет коммит к задаче.

### 5. Запушить ветку

```bash
git push origin 26.100/feature/bs.atroshenko/PAY-1238765
```

### 6. Создать Merge Request

Открой GitLab и создай MR из своей ветки в релизную ветку `rc/26.100`.

В описании MR обязательно укажи:
- Ссылку на задачу в Jira
- Что было сделано (кратко)
- Как тестировал
- Скриншоты, если менял UI

Назначь ревьюеров: минимум **2 аппрува** от коллег из команды. Один из ревьюеров должен быть senior или lead.

### 7. Прокинуть изменения в следующие релизные ветки

**Это критически важный шаг, который часто забывают.**

Если после `rc/26.100` существуют более поздние релизные ветки (`rc/26.200`, `rc/26.300`), твои изменения должны попасть и в них. Иначе в следующем релизе твой фикс или фича пропадёт.

Порядок действий:

```bash
# После мержа в rc/26.100 создай MR в следующие ветки:
# rc/26.100 → rc/26.200
# rc/26.200 → rc/26.300
# и так далее по цепочке
```

В GitLab это делается через интерфейс: после мержа основного MR нажми **Create merge request** и выбери target branch = `rc/26.200`. Повтори для каждой последующей ветки.

Если при мерже в следующую ветку возникли конфликты — разреши их локально:

```bash
git checkout rc/26.200
git pull origin rc/26.200
git merge rc/26.100
# разреши конфликты
git push origin rc/26.200
```

**Не игнорируй конфликты** и не делай `--force`. Если не уверен как разрешить — позови автора конфликтующего кода.

### Визуально весь процесс

```
rc/26.100 ──────────────────────────────●── (мерж твоего MR)
  │                                     │
  └─ 26.100/feature/bs.atroshenko/PAY-1238765
                                        │
rc/26.200 ──────────────────────────────●── (прокидка MR)
                                        │
rc/26.300 ──────────────────────────────●── (прокидка MR)
```

## Правила

- **Никогда не пушь напрямую** в релизные ветки. Только через MR с ревью.
- **Никогда не делай `git push --force`** в общие ветки. В своей личной ветке — можно (например, после `rebase`).
- **Один MR = одна задача Jira.** Не смешивай несколько задач в одном MR.
- **Удаляй свою ветку** после мержа. В GitLab стоит галочка «Delete source branch» — она включена по умолчанию, не снимай её.
- **Rebase vs Merge.** В FinBridge используется **merge-коммит** (стратегия `--no-ff`). Rebase разрешён только в личной ветке до первого ревью. После того как ревьюер начал смотреть код — только merge.

## Хотфиксы

Если нужно срочно починить что-то на проде:

1. Определи в какой релизной ветке находится текущий прод (информация в канале **#releases**).
2. Создай ветку от этой релизной ветки с типом `hotfix`.
3. Сделай MR с пометкой 🔥 **HOTFIX** в заголовке — такие MR ревьюятся в приоритете.
4. После мержа обязательно прокинь во все последующие релизные ветки.
5. Напиши в **#releases** что хотфикс замержен и готов к деплою.

## Полезные алиасы

Добавь в `~/.gitconfig` для ускорения работы:

```ini
[alias]
    rc = "!f() { git checkout rc/$1 && git pull origin rc/$1; }; f"
    nb = "!f() { git checkout -b $1; }; f"
    st = status --short
    lg = log --oneline --graph --decorate -15
    amend = commit --amend --no-edit
```

Использование:
```bash
git rc 26.100          # переключиться на релизную ветку и обновить
git nb 26.100/feature/bs.atroshenko/PAY-123  # создать ветку
git lg                 # компактная история
```

По вопросам Git Flow — пиши в **#engineering** или своему тимлиду.
